# microbe-foundation Benchmark Protocol

How to submit a new entry to the microbe-foundation leaderboard.

The benchmark exists to compare microbial genome encoders (KO presence, ESM-2 mean-pool, Bacformer, BacPT, MicroGenomer, gLM2, Evo 2, your own model — anything) on a unified 21-head trait surface with family-held-out evaluation. The defining property: **whole taxonomic families never span the train/val/test boundary**, so reported test scores measure cross-family generalization, not within-family memorization.

## The protocol

### 1. Use the official splits

```bash
python fetch_bacdive.py && python parse_bacdive.py && python splits.py
```

Or use the released `data/splits.parquet` from a tagged release (recommended for direct comparability — the splits are pinned with seed 42).

**Primary protocol: family-held-out.** Genus- and species-held-out are released for ablations but the primary leaderboard ranks on family.

### 2. Compute genome features

A valid feature matrix is one numpy `.npz` with:

| Key | Dtype | Shape | Notes |
|---|---|---|---|
| `bacdive_ids` | int64 | [N] | Maps each row to a BacDive strain |
| `features` | float32 | [N, D] | Per-genome feature vector |

**Allowed signals:** anything derived from the genome sequence (DNA, predicted proteins, gene annotations, KEGG modules, AMR catalogs, phylogenetic placement). 

**Not allowed:** any BacDive label fields (gram_stain, oxygen_tolerance, biosafety_level, etc.) — those are what we're predicting. Using GTDB taxonomy strings as features is also disallowed because the splits already key on family.

### 3. Train + evaluate

Use the released model.py for fair comparison:

```bash
python model.py \
    --features data/<your_features>.npz \
    --split-level family \
    --epochs 30 \
    --save-metrics runs/<your_method>.json \
    --run-name <your_method>
```

Or run your own training code, but **report on the exact same test-set strains** (read `splits.parquet` and filter to `family_split == "test"`).

### 4. Report

Each submission produces a JSON in the schema model.py emits:

```json
{
  "run_name": "esm2-150M",
  "split_level": "family",
  "epochs": 30,
  "n_params": 339000,
  "feature_dim": 640,
  "n_train": 15942,
  "n_val": 1993,
  "n_test": 1993,
  "per_head": {
    "gram_stain": {"metric_kind": "acc", "score": 0.94, "head_type": "multiclass", "head_size": 3},
    ...
  }
}
```

Per-head metrics:
- **binary, multiclass** → accuracy
- **multilabel** → F1 (macro across vocab)
- **regression_vector** (FAME) → RMSE on the normalized vector

### 5. Submit

Open a PR adding your `runs/<your_method>.json` and your code (or a link to your repo + commit SHA). The PR template asks for:

- One-paragraph description of your method
- Reproducibility commands (or repo link)
- Compute used (GPU type, hours)
- Any deviations from the protocol (e.g., different feature normalization, extra augmentation) — disclosed clearly

The leaderboard (`paper/tables/07_leaderboard.md` generated by `leaderboard.py`) ranks methods per-head and overall.

## What's special about each head

- **Cultivation medium** uses the implicit-negative assumption (strains with *any* medium documented count unlisted vocab items as negative). This is conventional in tag-prediction benchmarks but is a modeling choice — note in your submission if you use a different convention.
- **Pathogenicity** derives negatives from BSL-1 (German biosafety classification). Acknowledged limitation; we cannot tell "non-pathogen" from "undocumented" without this proxy.
- **FAME** is regression on a 30-dimensional normalized vector. Loss is masked-MSE on reported FAMEs only. **No prior published predictor exists** — your submission establishes the baseline.

## Versioning

The benchmark is currently `v0.1.0` (the schema version in `trait_schema.json`). Future versions may add heads (polar lipids, quinones once IJSEM PDF-mining lands), revise vocabularies, or change splits. Submissions should always cite the schema version they target so the leaderboard can be sliced cleanly.

## Rules summary

1. ✅ Use any sequence-derived feature
2. ✅ Use any model architecture and training procedure
3. ✅ Pretrain on external sequence corpora (UniRef, GTDB, MGnify, etc.)
4. ❌ Don't use BacDive label fields as features
5. ❌ Don't use family/genus name strings as features
6. ❌ Don't fine-tune on test-set strains
7. ✅ Report any deviation from the released model.py — transparency over conformity
